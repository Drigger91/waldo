# 2026-07-20

Versioning defaults closed out; then a long look downstream — how versioning meets
WAL/LSM, why SSTables are *not* next, where background work belongs, and a
re-factoring of the product: waldo stays a core engine, the smart layer becomes a
separate project.

## Versioning defaults: 5, no cap (closes the 07-14 amendment)

`MaxVersions` unset → `DefaultMaxVersions` (5); `1` reproduces the old latest-only
cache; anything higher is accepted. Briefly considered a hard ceiling (max 5, max
10) and dropped it — a fixed cap is arbitrary, can't express "keep every revision of
this prompt," and is exactly what MVCC (04) would have to delete, since chains there
are bounded by **the oldest live read seq**, not a constant.

Not the interesting part of the milestone and treated as such. The interesting part
is the trim in `pushVersion`: dropping the front of a slice without leaving the
backing array aliasing the dropped element (`e.versions[1:]` is right for *length*,
wrong for *lifetime* — invisible with `int`, a quiet leak with `[]byte` prompts).

## Versioning composes with WAL/LSM — `seq` is the shared primitive

Asked whether versioning would survive adding a WAL and SSTables. It doesn't just
survive; it's the same design. In LevelDB/RocksDB the internal key is
`(user_key, seq, type)`, sorted by user key ascending and **seq descending** — so
the newest version sorts first and a read is "seek, take the first record." That is
the `entry.versions` chain, flattened into a sorted file.

So the store-level `seq` does quadruple duty: version identity (02), log ordering
and replay idempotence (03), snapshot read points (04), and SSTable internal-key
ordering (if ever). Building it now under the write lock was the right call.

**What changes shape:** version trimming moves from *write time* to *compaction
time*. Old versions in an LSM live in immutable on-disk files; you don't rewrite a
file to drop one record. "Keep last N" becomes a compaction predicate, not a `Set`
predicate — and compaction is also where tombstones finally take effect (a tombstone
can't be dropped until it has outlived every older version of its key).

## SSTables are *not* needed next — but not for the reason first given

First reasoning was "we're a KV store maintaining versions ourselves, so we don't
need them." That's wrong and worth recording as wrong: versions are in-memory and
die with the process; SSTables are about what survives it and what happens when data
outgrows RAM. Orthogonal. Every LSM maintains versions itself *too*.

What SSTables actually buy, none of which versioning provides:
1. **Dataset larger than memory** — the main one.
2. **Sorted on-disk layout** → range scans / ordered iteration. A `map[K]entry[V]`
   cannot scan a range at any price.
3. **Bounded recovery time.**

(3) is the one that bites in 03: a WAL-only store replays the *entire log* at
startup, and the log grows forever because nothing truncates it. So **03 is really
WAL + a checkpoint**, and the choice is:

- **Periodic snapshot** — serialize the store, fsync, truncate the WAL. Simple;
  requires the dataset to fit in memory.
- **Memtable + SSTable flush** — the LSM answer. Levels, compaction, bloom filters,
  an on-disk format. Needed *only if* the data doesn't fit in memory.

**Decision: snapshot.** waldo has `MaxEntries` + LRU, so it is bounded in memory *by
construction* — the problem SSTables solve is one eviction already defines away.
Building an LSM under a cache that discards on a count budget is machinery with
nothing to do.

**The caveat that makes this conditional:** that's true of waldo-*the-cache*. If
agent memory pushes toward waldo-*the-database* (retain everything, never evict,
query by range), `MaxEntries` goes away, the in-memory bound goes with it, and
SSTables return immediately. **"No SSTables" follows from "waldo is a bounded
cache," and is only as settled as that identity.** ← still open.

## Where background work belongs

Considered a background sweeper to trim stale versions instead of trimming inline.
Rejected for the trim, kept for its neighbours. The rule:

> **Inline when the work is proportional to the write. Background when it's
> proportional to the data.**

Against a sweeper for trimming:
- `Set` already holds the write lock and the entry — the drop is O(1) on a slice of
  ≤5. A sweeper must *re-acquire* that lock, contending with live traffic, to do
  work that was free.
- Finding what to trim means scanning the map: O(keys) under the lock, a
  stop-the-world pause scaling with store size.
- `MaxVersions` stops being an invariant. Today "≤5 versions" holds at every
  instant; with a sweeper it holds *on average*, and a hot key between sweeps grows
  unbounded — defeating the memory bound the cap exists for. **Bounds that hold
  eventually aren't bounds.**
- waldo is an *embeddable library*. Spawning a goroutine in `New` imposes a
  lifecycle the caller didn't ask for (`Close()`, leaks if forgotten).
  `hashicorp/golang-lru`, the stated reference point, has no background anything.
  (Cost drops once the WAL forces a `Close()` regardless.)

Where background work *is* right:
- **Compaction** — mandatory in the SSTable world; can't be done inline. The
  instinct was correct, just aimed at 04/05's problem rather than 02's.
- **TTL expiry** — genuinely needs a sweeper, because *nothing triggers it*. A
  chain over-fills only when written, so the write can fix it; a key expires when
  nothing happens. Steal Redis's shape: lazy expiry on access **plus** an active
  cycle that samples ~20 random keys and repeats while the hit rate stays high —
  bounded work per tick, no full-scan pause. Likely its own milestone.

## Product re-factoring: core engine + separate utility project

Recorded here because it was never written down (07-14 has "core-first, then thin AI
faces", but those faces are *in-repo*; and fuzzy search appears only as deferred
*within* waldo).

The shape: **waldo stays a core storage engine.** The smart layer — e.g. a Python
fuzzy/semantic search + context-management tool sitting between agents and CLIs —
becomes a **separate project** consuming waldo over a network face.

Why it's better than what's on record:
- It **settles the cache-vs-database tension**. waldo keeps a boring, honest
  contract (versions, durability, eviction); ranking and recall live above it, free
  to be wrong and iterate without touching storage correctness.
- It makes the one-way dependency rule **physical** — a separate repo *cannot*
  accidentally import the core. No discipline required.

Two consequences signed up for:

1. **Index vs eviction.** Fuzzy/semantic search needs an index waldo shouldn't grow
   (inverted for string fuzz, vector for semantic). That index lives in the utility
   layer as a *derived copy* — and waldo **silently deletes things** under LRU. An
   external index over an evicting store is stale by construction: it returns hits
   for keys already dropped. So `OnEvict` is not a nice-to-have; it's the sync
   primitive the design rests on — **keep it through the WAL work**, and over a
   network face it likely needs to be a subscribable *stream*, not a callback.
2. **Sequencing flips.** Python can't link a Go library, so the network face becomes
   a *prerequisite* for the other project existing at all — not the last milestone.
   If the utility layer is where we learn whether any of this is useful, waiting
   until after MVCC is a long time without feedback. A minimal gRPC face over
   `Get`/`Set`/`History` is plausibly worth pulling forward to right after the WAL.

## Identity resolved: **database**. And MVCC moves ahead of the WAL.

The open question above got answered by naming the actual learning goals —
**transactions, isolation levels, WAL recovery**. All three are database topics;
the cache framing was inherited from 01 and had outlived its usefulness.

There's also a hard technical reason, not just a preference: **eviction and snapshot
isolation are incompatible.** A transaction reading at seq 100 breaks if LRU drops
that key mid-transaction — the snapshot becomes unreadable and the isolation
guarantee is silently violated. Real MVCC engines bound retention by the **oldest
live reader**, never by recency or count. So `MaxEntries` isn't merely secondary
under transactions; it's *wrong*. Demoted to an opt-in non-transactional policy.

Pleasingly, the code already agreed: `MaxEntries: 0` means unlimited, so the
zero-value store was already a database and the cache was already the opt-in.
Nothing to change but the framing (and the stale doc comments in waldo.go).

**Sequencing swap — MVCC (was 04) now comes before WAL (was 03).** The hard part of
a WAL is *atomicity*, and atomicity only means anything once there's a transaction
boundary. A WAL over single-key writes is just a log: append, replay, done — the easy
40%. Transaction ids, commit records, and "which transactions does recovery replay
vs discard" exist only *because* transactions do; build the WAL first and it gets
rewritten when they arrive. MVCC is also a direct continuation of 02 ("keep last N"
→ "keep what live snapshots need"), while durability is genuinely orthogonal and
layers cleanly onto a correct in-memory engine — and in-memory MVCC is far cheaper
to iterate on (no fsync, no crash harness, just logic).

Sharding also slid *after* the durable format, for the same layering reason the
per-shard-`seq` argument exposed: shard against a format that doesn't exist yet and
the format ends up shaped by the concurrency strategy.

New order: **02 versioning → 03 MVCC/txns → 04 isolation levels → 05 WAL+recovery →
06 sharding → 07 faces.**

## Two additions to the roadmap that weren't in it

- **Crash testing (05).** "Replay rebuilds state" with no way to *prove* it is a WAL
  you don't know works. A harness that kills the process at random points mid-write
  and asserts recovery invariants is where the actual lessons live: `write` vs
  `fsync` and what the OS really promises, torn/partial trailing records, per-record
  checksums, replay idempotence.
- **Property-based testing (03 onward).** Random operation sequences run against a
  trivial reference model (plain map + slice, no concurrency, obviously correct),
  asserting the two agree. Finds what unit tests structurally cannot, and becomes
  indispensable at 04 where hand-written tests can't cover interleavings.

**How we'll know isolation was actually learned:** build the anomaly table as a test
suite rather than reading it — dirty read, non-repeatable read, phantom, lost update,
**write skew**. Write skew is the one to chase: the anomaly snapshot isolation does
*not* prevent (two txns read an overlapping set, write disjoint parts, both commit,
and a constraint spanning them breaks even though nothing conflicts per-key). Making
it happen under our own SI, then watching it vanish under serializable, is the real
lesson. That test file is the best artifact this repo can produce.

## Six bugs in `pushVersion` / `History` — one lesson wearing several hats

Five of the six were **Go value semantics**: map values are copies, slice headers are
copies, and reslicing shares memory. Worth naming as one idea, because they don't
feel related while you're hitting them.

**Map values are copies.** `vals, exists := s.items[key]` hands back a *copy* of the
`entry` struct; every mutation to `vals.versions` evaporated at return. The store
accepted writes and retained nothing. Note the language half-warns about this: the
direct form `s.items[key].versions = ...` is a **compile error** (map values aren't
addressable) — copying to a local silences the error without fixing the cause.

**Slice headers are copies.** In `History`, `versions := e.versions` copies the
header — pointer, len, cap — not the elements. So `slices.Reverse(versions)`
reordered *the store's own chain, in place*, and returned a live reference to it.

**Reslicing shares memory.** The trim used `vals.versions[1:]` with a comment claiming
that assigning it to a new variable avoided referencing the dropped element. It does
not. A reslice only advances the header into the **same backing array**; the dropped
element stays reachable and alive. Invisible with `int`, a genuine leak with `[]byte`
values — `len()` says 3 while the array still pins every version ever written.
Fix: `copy` to shift left, `clear` the freed tail slot, then reslice.

Plus two ordinary ones: `isNew` returned inverted (so `Set` took the wrong policy
branch every time), and the trim fired on `== maxVersions` instead of `>`, capping
chains at `maxVersions - 1`.

## The one worth remembering: `History` silently broke `Get`

`History` never touches `Get`. But by reversing the shared backing array, it moved the
newest version away from the index `latest()` reads — so **after one `History` call,
`Get` returned the oldest value**, and a second `History` call quietly undid the
first.

A read method corrupting a *different* read method's results, connected only through
a slice header that looked like a copy. That's the shape of bug to expect from here
on: not a wrong line, but two correct-looking pieces of code sharing memory neither
of them thinks it shares. MVCC (03) is entirely made of this — snapshots and live
chains referencing the same versions — so the reflex to build now is **"who else can
see this memory?"** before "is this logic right?"

## The deadlock came back

`pushVersion` re-locked `s.mu` while `Set` already held it — the *exact* bug recorded
in [2026-07-09](2026-07-09.md), in a function whose doc comment already said "Caller
MUST hold s.mu."

The lesson isn't "be careful," it's that **a convention living only in a doc comment
is not a safeguard.** Renamed to `pushVersionLocked`: the `Locked` suffix is Go's
convention for helpers that assume the mutex is held, and unlike prose it's visible
at the call site. The compiler still won't enforce it — but the reader can't miss it.
Apply the same suffix to every locked helper as 03 adds them.

## The tests passed by luck of ordering

`TestStore_Versioning` did catch the `History` bug — but only because its assertions
happen to call `History` before the later `Get`s. Reorder the same assertions and it
goes green with the store silently corrupt.

That's the sharpest argument yet for the **property test** in
[plans/03-mvcc.md](../plans/03-mvcc.md): random operation sequences against a trivial
reference model would have found this on roughly the first run, independent of how
the assertions happened to be written. Hand-written tests verify the orderings you
thought of. This bug lived in one you didn't.

## Open questions

- **Does WAL replay restore full version history, or only the latest value per
  key?** Lands at 05; changes the log record format. Lean: log the *version*
  (`seq, key, value, ts`) — costs nothing extra now, and discarding history at replay
  is easy where inventing it later is impossible.
- **Does eviction survive in the API at all** once transactions land, or split into a
  separate non-transactional mode? Revisit at the end of 03.
