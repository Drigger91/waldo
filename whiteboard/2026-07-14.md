# 2026-07-14

Direction crystallised, and the versioning (02) design settled.

---

## Direction: core-first, then thin AI faces

waldo becomes a versioned, durable, embeddable KV **engine first**; then thin
network **faces** (MCP / gRPC / REST) expose it for AI use cases — prompt
versioning, response caching (TTL), agent memory.

Rules that keep it honest:
- **Dependency points one way:** faces import the core; the core *never* imports a
  face (no `http`/`grpc`/`mcp` types in the store package). That's what makes "swap
  the face, core intact" real instead of aspirational.
- **The face adds zero storage logic** — it translates a request into a core call
  and marshals the result back. Caching / versioning / expiry belong in the core.
- **Build order:** core features first; a face only once there's something worth
  exposing.

Honest caveat, on the record: as a *product* the moat is thin (Redis + a schema,
CLI tools growing their own memory, vector DBs own semantic recall). As a *learning
project + personal tool + portfolio piece*, it's strong. The product *motivates* the
roadmap; it does not reorder it around wrappers. Fuzzy/semantic key search is
deferred — and it's a different discipline (vector search, not string fuzz);
reconsider it only if agent-memory becomes the real driver.

Execution is now tracked in `plans/` (living checklists); this journal stays the
append-only *why*.

## Versioning (02) design — decided

Keep the last N versions per key. The incremental on-ramp to MVCC (04).

Data model:
```go
type Version[V any] struct { Seq uint64; Value V; Ts int64 }
type entry[V any]   struct { versions []Version[V] } // oldest-first, len ≤ N
```

Decisions:
1. **Unified Store, `MaxVersions` (default 1).** One store type; N=1 behaves exactly
   like today's cache, N>1 opts into history. No fork — progressive capability.
2. **`History(key) []Version[V]`**, newest-first, exposing `Seq` + `Ts` (not just
   values) — needed for "prompt v5", ordering, and it seeds MVCC. Returns a **copy**,
   never the internal slice (callers could mutate it).
3. **`Delete` drops the whole chain** (key + all versions). Simple and correct
   pre-MVCC; 04 will upgrade it to a tombstone version.

Settled by their nature (not debated): one key = one cache entry, so **LRU + the
count budget still key on the logical key**; no snapshot/read-at-seq yet (that's
MVCC); reading history counts as an access (bumps recency).

**Amended, same day — default is 5, not 1, and there is no cap.** Decision 1 above
said "default 1, N>1 opts into history." Reversed while writing the skeleton: waldo
is a *versioned* store, so the zero value of `Options` should say so — unset →
`DefaultMaxVersions` (5), and `MaxVersions: 1` is how you ask for the old
latest-only cache. Then briefly considered a hard ceiling (max 5, max 10) and
dropped it: a fixed cap is arbitrary, can't express "keep every revision of this
prompt," and is precisely the thing 04 would have to delete, since MVCC bounds
chains by **the oldest live read seq**, not by a constant. Building a wall now to
demolish it in two milestones is not a design.

Costs, on the record: `Options{}` now holds 5× the values it did before, and
`MaxVersions: 0` can no longer mean "unlimited" — that needs a sentinel (`-1`)
whenever MVCC wants it. Both are cheap; the version-count knob was never the
interesting part of this milestone. The trim in `pushVersion` is — dropping the
front of a slice while not leaving the backing array aliasing the dropped element.

`seq` is a store-level monotonic counter bumped **under the write lock** — no atomics
yet, since every mutation already holds the lock. Atomics arrive with lock-free MVCC
reads.

**Why it generalises to MVCC:** "keep last N" → "keep every version a live snapshot
can see"; "drop the oldest" → GC by the oldest live read seq. Same chain, richer
visibility rule.
