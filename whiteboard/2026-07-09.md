## Phase 1 design: store, eviction & concurrency (the plan)

Phase 1 is ~200 lines sitting on a handful of design questions. Where they landed:

### Store / eviction split — "who owns the bytes?"

- **A — policy tracks keys only; store owns values.** Two maps keyed by `K` (key
  stored twice), but the policy is tiny, `V`-agnostic, and swapping LRU→FIFO→LFU is
  swapping one small object. The cleanest "eviction is an interface."
- **B — policy owns the whole entry (K+V).** One map, one lookup — but the policy
  *is* the storage engine, generic over `K` and `V`. This is what
  `hashicorp/golang-lru` actually is.
- **C — one map, intrusive list.** Store owns `map[K]*entry` with an opaque handle;
  one lookup, still swappable, at the cost of coupling. The eventual optimization.

**Decision: start with A**; build **C later and benchmark the delta**.

**Refinement while coding:** the policy ended up **cost-free** (`Add(key)`, no
`cost`). The store owns accounting and drives eviction; the policy only answers
"which key is next?" → clean split: **policy = ordering, store = accounting**.

### Capacity

Started with a dual `MaxEntries` + `MaxBytes` plan, then **parked the byte budget**:
sizing an arbitrary/heterogeneous value is its own problem (and with `V=any`,
Redis-style, it's genuinely hard — Redis tracks memory via its own encodings).
First cut is **count-only** (`MaxEntries`). Byte budget returns as a later increment.

### The concurrency arc (each step benchmarked against the last)

1. **Single `RWMutex`, full `Lock` in `Get`** — honest baseline; the RWMutex buys
   nothing yet, and that's the point.
2. **Sharded locks** — `shard = hash(key) % N`; each shard owns its own mutex + LRU
   list. The *bounded* form of "lock per key."
3. **Timestamp + sampling** — kill the shared list entirely (see below).

## Why `Get` can't use `RLock` — integrity, not accuracy

I kept wanting `Get` to take `RLock` (readers shouldn't block each other). Wrong,
and the *why* is the whole Phase 1 concurrency lesson. Two **separate** problems; I
was reasoning about the wrong one:

- **Problem A — eviction accuracy.** A lost/raced recency bump makes eviction
  slightly less accurate. *Tolerant* — approximate LRU is fine; real caches embrace it.
- **Problem B — data-structure integrity.** `Touch` → `list.MoveToFront` rewires ~6
  pointers. `RLock` lets many goroutines run at once, so two `Touch`es splice the
  same list simultaneously → orphaned nodes, cycles, list/map desync; a later
  `Evict` then walks a corrupt list and hangs or panics. *Not* tolerant.

**The lock exists for B, not A.** The clincher: this corruption happens even with a
huge cache and **zero evictions** — an all-`Get` workload still runs concurrent
`MoveToFront`. Capacity is irrelevant; the damage is to the *structure*, not the
eviction *decision*.

Rule, no exceptions: concurrent writes to the same non-atomic memory = UB. A
multi-pointer splice needs *exclusive* access → a mutating `Get` needs `Lock`.

**When `RLock`/lock-free *is* allowed:** when the per-`Get` update is a single
*atomic word* write, not a splice — a timestamp, `atomic.StoreInt64(&e.atime, clock)`.
That's a *different data structure* (timestamps + sampling), and it's where
"approximate eviction is fine" (Problem A) finally justifies the sampling.

**Backpressure aside:** you *can* move recency off the read path (fire a "touched K"
signal, apply it in a background goroutine). The real question is where backpressure
lands when signals outpace the applier: block the reader (reinvents the lock),
**lossy drop** (fine *iff* recency is droppable — Ristretto does this), or unbounded
queue (OOM trap). Recency is best-effort → lossy is legit. Contrast Phase 2 group
commit: durability is a *promise* → must block/reject, never drop. Same machinery,
opposite policy. (A Go `chan` is itself mutex-backed; per-`Get` sends can cost more
than the lock — Ristretto wins via lossy + batching, not "streaming".)

## Gotcha: Go mutexes are not reentrant — the eviction deadlock

First cut of `Set` took the write lock and then called `evict()`, which **also**
locked:

```go
func (s *store) Set(...) { s.mu.Lock(); defer s.mu.Unlock(); ... s.evict() }
func (s *store) evict()  { s.mu.Lock(); defer s.mu.Unlock(); ... } // ← same lock, same goroutine
```

Go's `sync.Mutex`/`RWMutex` is **not reentrant/recursive** (unlike Java's
`synchronized` or a pthreads recursive mutex). A goroutine that already holds the
lock and tries to take it again blocks **on itself, forever**. Single-goroutine, the
runtime catches it: `fatal error: all goroutines are asleep - deadlock!`; under many
goroutines it just hangs until the test timeout. It only fires once eviction
actually triggers, so it hid behind the unbounded default.

**Rule:** pick *one* layer that owns locking — here the public methods
(`Get`/`Set`/`Delete`) — and have private helpers **assume** the lock is held
(document `// Caller MUST hold s.mu`). Never lock in both. Fix was to delete
`evict()`'s own `Lock`/`Unlock`; `Set` holds the lock across the whole operation.

(Go omits recursive mutexes on purpose: they encourage fuzzy lock scoping and hide
exactly this class of bug. The discipline of "one owner of the lock" is the feature.)

## Gotcha: evict-before-add → capacity floats at `maxItems + 1`

`overCapacity()` is `len(items) > maxItems`, and it was checked *before* inserting
the new key: with `maxItems=3` and 3 items, `3 > 3` is false → no evict → insert →
`len 4`. It only trims on the *next* `Set`, stabilizing at 4, not 3.

Fix: **add first, then trim** — `s.items[key]=…; policy.Add(key); s.evict()` — and
make `evict()` a `for` loop (`for s.overCapacity() { … }`), not a single `if`. Now
the store sits at exactly `maxItems`, and it's robust even if `maxItems` is lowered
later or we're ever over by more than one.

## Baseline captured (keep the numbers)

First cut is green under `go test -race ./...`. `BenchmarkStore_ParallelGet` on an
Apple M2, full-`Lock`-in-`Get`:

| cores | ns/op |
|---|---|
| 1 | 33.6 |
| 2 | 57.8 |
| 4 | 105.4 |
| 8 | 124.9 |

**Negative scaling** — more cores = slower, because every `Get` serializes on the
one mutex and bounces its cache line. This is the number sharding has to beat.
(`BenchmarkStore_Set` ≈ 29.6 ns/op, 0 allocs — single-goroutine, so flat.)

**Next:** sharded locks → re-run this benchmark → expect it to *scale up* instead
of down.
